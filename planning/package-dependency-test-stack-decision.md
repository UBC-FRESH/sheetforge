# Package, Dependency, And Test Stack Decision

Date: 2026-06-19

## Purpose

This note records the Phase 5 decision for package layout, dependency management, test framework, and fixture location.

The decision is intentionally narrow: introduce enough durable structure to turn the proven scratch prototypes into tested package code, without adding CLI, CI, or broad framework commitments yet.

## Package Layout

Use a standard `src/` layout:

```text
src/
  sheetforge/
    __init__.py
    ...
tests/
  ...
```

Rationale:

- keeps import behavior honest during tests;
- works with standard Python packaging tools;
- leaves room for extraction, IR, code-generation, validation, and diagnostics modules;
- avoids exposing ignored `tmp/` scratch assumptions as package code.

Initial package modules should remain minimal and should be introduced only when P5.5 defines the implementation bootstrap plan.

## Dependency Manager

Use `pyproject.toml` with standard `setuptools` build metadata for the first durable package slice.

Rationale:

- minimal and widely supported;
- no project-specific dependency manager lock-in yet;
- compatible with editable installs and `pytest`;
- sufficient for early package/test work.

Do not introduce Poetry, Hatch, PDM, uv lockfiles, or Conda environment files yet. Those can be revisited if dependency complexity grows.

## Runtime Dependencies

Initial runtime dependencies should be conservative:

- `openpyxl` for workbook extraction.

Keep these out of the default runtime dependency set for now:

- `formulas`, until the validation/oracle API boundary is finalized;
- `networkx`, until graph utilities need more than small local algorithms;
- `xlwings`, because Excel-backed validation is optional and platform-specific.

Use optional dependency groups when needed, for example:

```text
[validation]
formulas

[excel]
xlwings
```

The exact optional dependency names can be finalized when the package metadata is written.

## Test Framework

Use `pytest`.

Rationale:

- standard and lightweight;
- good fixture support;
- simple assertion style for prototype-derived behavior;
- works well with small generated files and JSON reports.

Avoid adding coverage, property-based testing, snapshot tooling, or test plugins until a concrete need appears.

## Fixture Location

Use tracked fixtures under:

```text
tests/fixtures/
```

Recommended first fixture assets:

- a tiny synthetic workbook, generated by a tracked fixture builder or stored directly if small;
- expected validation scenario JSON;
- expected output values for the baseline validation case;
- one intentional mismatch fixture or scenario once mismatch handling is implemented.

Do not commit private workbooks, large workbooks, or bulky generated artifacts.

## Generated Artifacts In Tests

Prefer regenerating derived artifacts during tests rather than committing generated code or generated reports.

Tracked fixtures should be source-like and compact. Generated model files, IR JSON, and validation reports can be written to pytest temporary directories.

## Phase 5 Implementation Boundary

This decision does not add package files yet.

P5.2 should define API boundaries first. P5.3 should define fixture and regression-test strategy. P5.5 should then decide the first implementation bootstrap plan and create the package skeleton if still warranted.

## Decisions Summary

- Package layout: `src/sheetforge`.
- Build metadata: `pyproject.toml` with `setuptools`.
- Test framework: `pytest`.
- Fixture root: `tests/fixtures`.
- Runtime dependency default: start with `openpyxl`.
- Optional validation dependency: likely `formulas`.
- Optional Excel dependency: likely `xlwings`, not default.
- CLI: deferred to P5.2/P5.5.
- CI: deferred to P5.4.
